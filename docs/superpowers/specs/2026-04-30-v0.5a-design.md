# MikroMCP v0.5a Design — Advanced Firewall, Policy Routing & Routing Protocols

**Date:** 2026-04-30  
**Milestone:** v0.5a (first sub-phase of v0.5)  
**Scope:** New tools only — Firewall Mangle, Address Lists, Policy Routing, Routing Protocols (read-only)

## Overview

v0.5a adds 10 new tools across four subsystems. No existing files are modified except `src/domain/tools/index.ts` (four new imports and spreads). All tools follow established patterns in the codebase.

## Tool Specifications

### 1. Mangle Tools (`mangle-tools.ts`)

#### `list_mangle_rules`

- **Path:** `GET /ip/firewall/mangle`
- **readOnlyHint:** `true` (auto-retry enabled)
- **Returns:** Rules in evaluation order
- **Optional filters:** `chain` (string), `action` (string), `disabled` (boolean)

#### `manage_mangle_rule`

- **readOnlyHint:** `false`, **destructiveHint:** `false`, **idempotentHint:** `true`
- **Idempotency key:** `comment` (required on all actions — same pattern as `manage_firewall_rule`)
- **Supports dry-run:** yes

**Input schema:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `routerId` | string | yes | Target router |
| `action` | `"add" \| "remove" \| "enable" \| "disable"` | yes | |
| `comment` | string | yes | Idempotency key |
| `chain` | string | on `add` | e.g. `"prerouting"`, `"forward"` |
| `dryRun` | boolean | no | Default `false` |
| `srcAddress` | string | no | Source IP/CIDR |
| `dstAddress` | string | no | Destination IP/CIDR |
| `srcAddressList` | string | no | Source address list name |
| `dstAddressList` | string | no | Destination address list name |
| `protocol` | string | no | e.g. `"tcp"`, `"udp"` |
| `srcPort` | string | no | Port or range |
| `dstPort` | string | no | Port or range |
| `inInterface` | string | no | Incoming interface |
| `outInterface` | string | no | Outgoing interface |
| `newRoutingMark` | string | no | Routing mark to set |
| `newConnectionMark` | string | no | Connection mark to set |
| `newDscpValue` | number | no | DSCP value to set |
| `passthrough` | boolean | no | Whether to continue matching |

**Idempotency behaviour:**

| Scenario | Result |
|---|---|
| `add` — comment not found | create, return `action: "created"` |
| `add` — comment found, same config | return `action: "already_exists"` |
| `add` — comment found, different config | throw `CONFLICT` with existing vs requested diff |
| `remove` — comment found | remove, return `action: "removed"` |
| `remove` — comment not found | return `action: "already_removed"` |
| `enable` / `disable` — comment found | update `disabled` field, return `action: "updated"` |
| `enable` / `disable` — comment not found | throw `NOT_FOUND` |

### 2. Address List Tools (`address-list-tools.ts`)

#### `list_address_list_entries`

- **Path:** `GET /ip/firewall/address-list`
- **readOnlyHint:** `true`
- **Returns:** All entries across all lists
- **Optional filters:** `list` (string — list name), `address` (string)

#### `manage_address_list_entry`

- **readOnlyHint:** `false`, **destructiveHint:** `false`, **idempotentHint:** `true`
- **Idempotency key:** `list` + `address` (composite)
- **Supports dry-run:** yes

**Input schema:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `routerId` | string | yes | |
| `action` | `"add" \| "remove"` | yes | |
| `list` | string | yes | Address list name |
| `address` | string | yes | IP or CIDR |
| `comment` | string | no | |
| `timeout` | string | no | e.g. `"1d"`, `"2h30m"` — omit for permanent |
| `dryRun` | boolean | no | Default `false` |

**Idempotency behaviour:**

No `CONFLICT` case — entries have no config that can diverge, only presence/absence matters.

| Scenario | Result |
|---|---|
| `add` — list+address not found | create, return `action: "created"` |
| `add` — list+address found | return `action: "already_exists"` |
| `remove` — list+address found | remove, return `action: "removed"` |
| `remove` — list+address not found | return `action: "already_removed"` |

### 3. Policy Routing Tools (`policy-routing-tools.ts`)

#### `list_routing_rules`

- **Path:** `GET /routing/rule`
- **readOnlyHint:** `true`
- **Returns:** Rules in evaluation order
- **Optional filters:** `table` (string), `disabled` (boolean)

#### `manage_routing_rule`

- **readOnlyHint:** `false`, **destructiveHint:** `false`, **idempotentHint:** `true`
- **Idempotency key:** composite of `srcAddress` + `dstAddress` + `interface` + `table`
- **Supports dry-run:** yes

**Input schema:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `routerId` | string | yes | |
| `action` | `"add" \| "remove" \| "enable" \| "disable"` | yes | |
| `table` | string | yes | Routing table to use (the action); required on all actions as it is part of the composite idempotency key |
| `srcAddress` | string | no | Source CIDR |
| `dstAddress` | string | no | Destination CIDR |
| `interface` | string | no | Incoming interface |
| `priority` | number | no | 0–4294967295 |
| `dryRun` | boolean | no | Default `false` |

On `add`, at least one of `srcAddress`, `dstAddress`, or `interface` must be provided (validated in schema with `.refine()`).

**Idempotency behaviour:**

The composite key IS the full config, so there is no `CONFLICT` case — a rule either matches all four fields or it doesn't.

| Scenario | Result |
|---|---|
| `add` — key not found | create, return `action: "created"` |
| `add` — key found | return `action: "already_exists"` |
| `remove` — key found | remove, return `action: "removed"` |
| `remove` — key not found | return `action: "already_removed"` |
| `enable` / `disable` — key found | update `disabled`, return `action: "updated"` |
| `enable` / `disable` — key not found | throw `NOT_FOUND` |

#### `list_routing_tables`

- **Path:** `GET /routing/table`
- **readOnlyHint:** `true`
- **Returns:** All custom routing tables

#### `manage_routing_table`

- **readOnlyHint:** `false`, **destructiveHint:** `false`, **idempotentHint:** `true`
- **Idempotency key:** `name`
- **Supports dry-run:** yes

**Input schema:**

| Field | Type | Required | Notes |
|---|---|---|---|
| `routerId` | string | yes | |
| `action` | `"add" \| "remove"` | yes | |
| `name` | string | yes | Table name |
| `fib` | boolean | no | Sync with FIB, default `false` |
| `dryRun` | boolean | no | Default `false` |

**Idempotency behaviour:**

| Scenario | Result |
|---|---|
| `add` — name not found | create, return `action: "created"` |
| `add` — name found | return `action: "already_exists"` |
| `remove` — name found | remove, return `action: "removed"` |
| `remove` — name not found | return `action: "already_removed"` |

### 4. Routing Protocol Tools (`routing-protocol-tools.ts`)

Both tools are read-only with no write counterparts in v0.5a. ROS 7+ only.

#### `list_bgp_peers`

- **Path:** `GET /routing/bgp/session`
- **readOnlyHint:** `true`
- **Returns:** Active BGP sessions with: remote address, remote AS, local address, state (`established` / `active` / `idle` / etc.), uptime, prefixes received, prefixes sent, last error
- **Optional filter:** `state` (string)

#### `list_ospf_neighbors`

- **Path:** `GET /routing/ospf/neighbor`
- **readOnlyHint:** `true`
- **Returns:** Neighbors with: neighbor ID, interface, state (`full` / `2-way` / etc.), DR address, BDR address, priority, uptime
- **Optional filter:** `state` (string)

## Test Coverage

Each tool file has a corresponding test file following the standard groups:

1. **metadata** — correct tool count, names, `readOnlyHint` / `destructiveHint` annotations
2. **input schema** — valid input with defaults, rejection of extra fields (`.strict()`), rejection of out-of-range values, required-field enforcement, `.refine()` validation (routing rule at-least-one-of check)
3. **handler: happy path** — returns expected `structuredContent` shape
4. **handler: idempotency** — `already_exists`, `already_removed` as appropriate per tool
5. **handler: conflict** — `CONFLICT` error for mangle rules (comment match, config mismatch)
6. **handler: dry-run** — returns `action: "dry_run"` without calling `create` or `remove`
7. **handler: not-found** — `NOT_FOUND` thrown for enable/disable when key not found (mangle, routing rule)

All tests use the `makeContext` mock pattern. No integration tests.
