## 2. Configuration: `run_command` Allow/Deny

### Resolution order (highest to lowest priority)

1. Per-router YAML (`cmdAllow` / `cmdDeny` arrays under the router entry)
2. Global env vars (`MIKROMCP_CMD_ALLOW` / `MIKROMCP_CMD_DENY`, comma-separated glob patterns)
3. Built-in defaults (deny list below; allow list empty = everything allowed)

### Config schema addition (`routers.yaml`)

```yaml
routers:
  - id: my-router
    host: 192.168.1.1
    # ...
    cmdAllow:         # optional; if set, command must match at least one pattern
      - "/ip/route/*"
    cmdDeny:          # optional; merged additively with global and built-in deny lists
      - "/system/backup/*"
```

### Built-in default deny list

Patterns are glob-matched case-insensitively against the full command string. The operator's deny list is **additive** ‚Äî it merges with, not replaces, these defaults.

- `/system shutdown*`
- `/system reboot*` ‚Äî use the dedicated `reboot` tool instead
- `/system reset-configuration*`
- `/user set*`
- `/user add*`
- `/user remove*`
- `/ip service set*` ‚Äî could lock out management access
- `/certificate*private-key*` ‚Äî key export
- `/system package uninstall*`
- `/system routerboard upgrade*`

### Evaluation logic

1. Check command against merged deny list (built-in + global env + per-router YAML). If matched ‚Üí reject with `ErrorCategory.VALIDATION`, `suggestedAction` names the dedicated tool where one exists.
2. If an allow list is configured (global env or per-router), check command matches at least one pattern. If not ‚Üí reject with `VALIDATION`.
3. If `dryRun: true` ‚Üí return preview without executing.
4. Execute and return output, capped at 4000 characters (truncated with warning if exceeded).

## 3. New Tools

### 3.1 `ping` (`diagnostic-tools.ts`)

- **RouterOS path:** `tool/ping` (POST)
- **Annotations:** `readOnlyHint: true`, auto-retry enabled
- **Input:**
  - `routerId: string`
  - `address: string` ‚Äî target IP or hostname
  - `count: number` (1‚Äì20, default 4)
  - `size: number` ‚Äî packet size in bytes (14‚Äì65535, default 56)
  - `routingTable?: string` ‚Äî optional routing table to use
- **Output:** per-packet RTT list, min/avg/max RTT, packet loss %

### 3.2 `traceroute` (`diagnostic-tools.ts`)

- **RouterOS path:** `tool/traceroute` (POST)
- **Annotations:** `readOnlyHint: true`, auto-retry enabled
- **Input:**
  - `routerId: string`
  - `address: string`
  - `count: number` (1‚Äì5, default 3) ‚Äî probes per hop
  - `maxHops: number` (1‚Äì30, default 15)
- **Output:** ordered hop list, each with address and RTT

### 3.3 `torch` (`diagnostic-tools.ts`)

- **RouterOS path:** `tool/torch` (POST)
- **Annotations:** `readOnlyHint: true`, auto-retry enabled
- **Note:** the tool call blocks synchronously for `duration` seconds ‚Äî no async polling needed
- **Input:**
  - `routerId: string`
  - `interface: string` ‚Äî required; interface name to monitor
  - `duration: number` (1‚Äì30 seconds, default 5)
  - `srcAddress?: string` ‚Äî optional source address filter
  - `dstAddress?: string` ‚Äî optional destination address filter
- **Output:** top flows by bytes during the capture window (tx-bytes, rx-bytes, src, dst)

### 3.4 `get_log` (`diagnostic-tools.ts`)

- **RouterOS path:** `log` (GET)
- **Annotations:** `readOnlyHint: true`, auto-retry enabled
- **Input:**
  - `routerId: string`
  - `limit: number` (1‚Äì500, default 100)
  - `offset: number` (default 0)
  - `topics?: string[]` ‚Äî filter entries whose topics include any of these strings (e.g. `["firewall", "dhcp"]`)
  - `prefix?: string` ‚Äî substring match on the log message
  - `sinceMinutes?: number` (1‚Äì1440) ‚Äî only return entries from the last N minutes; requires client-side RouterOS timestamp parsing
- **Notes:** RouterOS log timestamps use two formats: `"jan/02 12:34:56"` for older entries and `"00:00:05"` (elapsed since boot) for very recent entries. The handler parses both, falls back to including the entry if parsing fails (conservative).

### 3.5 `get_system_clock` (`system-ops-tools.ts`)

- **RouterOS path:** `system/clock` (GET)
- **Annotations:** `readOnlyHint: true`, auto-retry enabled
- **Input:** `routerId: string`
- **Output:** `date`, `time`, `time-zone-name`, `time-zone-autodetect`
- **Note:** `get_system_status` can also return clock as a section; this tool is a focused single-purpose read.

### 3.6 `set_system_clock` (`system-ops-tools.ts`)

- **RouterOS path:** `system/clock` (PATCH)
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`
- **Input:**
  - `routerId: string`
  - `date?: string` ‚Äî RouterOS format `"jan/02/2006"`
  - `time?: string` ‚Äî RouterOS format `"15:04:05"`
  - `timeZoneName?: string` ‚Äî IANA timezone name (e.g. `"Europe/London"`)
  - `dryRun: boolean` (default false)
- **Idempotency:** reads current clock, returns `already_set` if values already match

### 3.7 `reboot` (`system-ops-tools.ts`)

- **RouterOS path:** `system/reboot` (POST)
- **Annotations:** `readOnlyHint: false`, `destructiveHint: true`, `idempotentHint: false`
- **Input:**
  - `routerId: string`
  - `delay: number` ‚Äî seconds before reboot (0‚Äì3600, default 0)
  - `dryRun: boolean` (default false)
- **Safety:** the built-in `run_command` deny list blocks `/system reboot*`, funneling all reboots through this tool with its explicit dry-run gate.

### 3.8 `run_command` (`system-ops-tools.ts`)

- **Transport:** SSH (not REST) ‚Äî RouterOS REST has no direct "run arbitrary command" endpoint; SSH is cleaner and leaves no temp artifacts on the router
- **SSH client:** use the `ssh2` npm package; connection parameters (host, port, username, password/key) sourced from the existing `RouterConfig` + resolved credentials ‚Äî same as the REST client
- **Execution:** open an SSH session, run the command as a single exec channel, collect stdout+stderr, close session
- **Config addition:** `src/types.ts` `RouterConfig` gains optional `sshPort: number` (default 22); `routers.yaml` schema updated accordingly
- **Annotations:** `readOnlyHint: false`, `destructiveHint: true`, `idempotentHint: false`
- **Input:**
  - `routerId: string`
  - `command: string` ‚Äî RouterOS console command
  - `dryRun: boolean` (default false)
- **Output:** command output string, capped at 4000 characters (truncated with notice if exceeded)
- **Guardrails:** deny/allow resolution as described in Section 2

## 4. Existing Tool Improvements

### `list_interfaces`

- Add `running?: boolean` ‚Äî when set, filters by the `running` field (`=== "true"` / `=== "false"`)
- Add `macAddress?: string` ‚Äî exact match on `mac-address` field (case-insensitive)

### `manage_ip_address`

- Fix network auto-calculation: only apply the plain-IP ‚Üí `/32` CIDR transform when the input has no prefix notation. When a prefix is present (e.g. `192.168.1.5/24`), pass `address` as-is and let RouterOS derive the network ‚Äî do not normalize the host bits.

### `manage_firewall_rule`

- Add `srcPort?: string` ‚Äî single port (`"80"`) or range (`"8000-9000"`); maps to `src-port`
- Add `dstPort?: string` ‚Äî same format; maps to `dst-port`
- Add `inInterface?: string` ‚Äî maps to `in-interface`
- Add `outInterface?: string` ‚Äî maps to `out-interface`
- Idempotency: these new fields are included in the conflict check ‚Äî if a rule with the same `comment` key exists but has different port/interface config, throw `ErrorCategory.CONFLICT` (not a silent update)

## 5. Error Handling

All handlers follow existing conventions:
- Body wrapped in `try/catch`; `enrichError(err, context)` in catch
- `MikroMCPError` re-thrown as-is; never double-wrapped
- Domain errors thrown as `new MikroMCPError(...)` directly

`run_command`-specific errors:
- Deny-list match ‚Üí `ErrorCategory.VALIDATION`, message names the matched pattern, `suggestedAction` names the dedicated tool if one exists
- Allow-list miss ‚Üí `ErrorCategory.VALIDATION`, message states the command is not in the configured allow list

Diagnostic tools (ping/traceroute/torch) treat partial results (e.g. 100% packet loss, unreachable hops) as valid responses ‚Äî not errors. Only network/auth failures become thrown errors.
