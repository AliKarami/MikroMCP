# v0.4 — Network Services: Design Spec

**Date:** 2026-04-28  
**Milestone:** v0.4  
**Goal:** WiFi, bridging, WireGuard, and DNS — the next most-configured subsystems after routing and firewall — preceded by a set of pre-existing infrastructure bug fixes.

---

## Overview

v0.4 is structured in six phases executed in order. Phases 1 and 6 are cross-cutting (infra fixes and docs); Phases 2–5 each deliver a self-contained tool subsystem. All new tools follow the established `ToolDefinition` pattern in `src/domain/tools/`.

### Decisions recorded

- WiFi targets `/interface/wifi` (RouterOS 7.13+). Older `/interface/wireless` support is deferred to a future milestone.
- `manage_dns_entry` supports A records only this milestone. CNAME/MX/other types are deferred.
- `manage_wireguard_peer` supports add, update, and remove (update patches only supplied fields).
- Infrastructure fixes and new tools ship in one plan; fixes are prerequisite to the tools.

### Deferred (noted in ROADMAP.md)

- WiFi support for older RouterOS (`/interface/wireless`)
- DNS multi-record types (CNAME, MX, TXT, etc.)

## Phase 1 — Infrastructure Fixes

Six isolated fixes, each independently testable.

### 1.1 Circuit breaker — ignore non-recoverable errors

**File:** `src/adapter/circuit-breaker.ts`

`onFailure()` currently increments on every thrown error. Fix: in `execute()`, catch the error and check if it is a `MikroMCPError` with category `VALIDATION`, `NOT_FOUND`, `CONFLICT`, or `COMMAND_DENIED`. If so, rethrow without calling `onFailure()`. Only `ROUTER_ERROR`, `ROUTER_UNREACHABLE`, `ROUTER_TIMEOUT`, and `ROUTER_BUSY` should trip the breaker.

```typescript
// in execute(), after catching:
if (
  error instanceof MikroMCPError &&
  NON_TRIPPING_CATEGORIES.has(error.category)
) {
  throw error; // don't count
}
this.onFailure();
throw error;
```

### 1.2 ZodError → VALIDATION in enrichError

**File:** `src/domain/errors/error-enricher.ts`

Add a branch before the HTTP-status check:

```typescript
import { ZodError } from "zod";
// ...
if (error instanceof ZodError) {
  return new MikroMCPError({
    category: ErrorCategory.VALIDATION,
    code: "VALIDATION_ERROR",
    message: "Invalid input parameters",
    details: buildDetails({ issues: error.issues }, context),
    recoverability: defaultRecoverability(ErrorCategory.VALIDATION),
    cause: error,
  });
}
```

### 1.3 Router registry — Zod schema validation at startup

**File:** `src/config/router-registry.ts`

Define a Zod schema for `RouterConfig` and validate the parsed YAML. On `ZodError`, log the issues and call `process.exit(1)` with a descriptive message. Missing file continues to warn + empty registry (existing behavior).

Schema covers: `host` (string, required), `port` (number, optional, default 443), `tls` (boolean, optional, default true), `envPrefix` (string, required), `tags` (string array, optional default `[]`), `sshPort` (number, optional).

### 1.4 Retry wraps circuit — restructure execution order

**File:** `src/mcp/tool-registry.ts`

Current order: `withRetry(() => cb.execute(() => handler()))` — each retry independently trips the circuit.

Flip to: `cb.execute(() => withRetry(() => handler()))` — the circuit wraps the whole retry sequence.

```typescript
const executeHandler = () =>
  cb!.execute(() =>
    tool.annotations.readOnlyHint
      ? withRetry(() => tool.handler(args, context), config.retry)
      : tool.handler(args, context),
  );
const result = await executeHandler();
```

### 1.5 IP/CIDR validation — stricter parser

**New file:** `src/domain/tools/net-utils.ts`

```typescript
export function isValidCidr(s: string): boolean {
  const [addr, prefix] = s.split("/");
  if (!addr || prefix === undefined) return false;
  const parts = addr.split(".");
  if (parts.length !== 4) return false;
  if (!parts.every((p) => /^\d{1,3}$/.test(p) && Number(p) <= 255)) return false;
  const pfx = Number(prefix);
  return Number.isInteger(pfx) && pfx >= 0 && pfx <= 32;
}
```

Replace the loose regex in `ip-tools.ts` and `route-tools.ts` with a `.refine(isValidCidr, "Invalid CIDR")` call on the address fields.

## Phase 2 — Bridge Tools

**New file:** `src/domain/tools/bridge-tools.ts`  
**RouterOS paths:** `interface/bridge`, `interface/bridge/port`

### `list_bridges`

- **Type:** read-only (`readOnlyHint: true`)
- **Input:** `routerId`, `limit` (default 100, max 500), `offset` (default 0)
- **Behavior:** Fetch all from `interface/bridge`. For each bridge, fetch `interface/bridge/port` filtered by `bridge` name to get port members. Return bridges with embedded `ports[]`.
- **Structured result:** `{ routerId, bridges: [{ ...bridgeRecord, ports: [...portRecords] }], total, hasMore, offset, limit }`

### `manage_bridge`

- **Type:** write (`readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: add/remove), `name`, `mtu` (optional, default 1500), `comment` (optional), `dryRun`
- **Idempotency key:** `name`
- **add:** Look up existing by name.
  - Same name, same MTU → `already_exists`
  - Same name, different MTU → `CONFLICT` with existing MTU in `details`
  - Not found → create
- **remove:** Look up by name. Not found → return `not_found` result (not an error). Found → delete.
- **dry-run:** Return diff without applying.

### `manage_bridge_port`

- **Type:** write (`readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: add/remove), `bridge`, `interface`, `dryRun`
- **Idempotency key:** `(bridge, interface)` pair
- **Prerequisite:** Verify bridge exists via `interface/bridge`; throw `NOT_FOUND` if missing.
- **add:** Check `interface/bridge/port` for existing (bridge, interface) pair.
  - Exists → `already_exists`
  - Not found → create
- **remove:** Check for existing pair. Not found → `not_found`. Found → delete.

## Phase 3 — WiFi Tools

**New file:** `src/domain/tools/wifi-tools.ts`  
**RouterOS paths:** `interface/wifi`, `interface/wifi/configuration`, `interface/wifi/registration-table`  
**Target:** RouterOS 7.13+ only.

### `list_wifi_interfaces`

- **Type:** read-only
- **Input:** `routerId`, `disabled` (optional boolean filter), `limit`, `offset`
- **Behavior:** Fetch all from `interface/wifi`. Fetch all from `interface/wifi/configuration`. Join by matching the interface's `configuration` field to the config's `name`. Return merged records.
- **Structured result:** `{ routerId, interfaces: [{ ...ifaceRecord, ssid, band, channel }], total, hasMore, offset, limit }`

### `list_wifi_clients`

- **Type:** read-only
- **Input:** `routerId`, `interface` (optional string filter), `limit`, `offset`
- **Behavior:** Fetch from `interface/wifi/registration-table`. If `interface` is provided, filter client-side. Fields: MAC, interface, signal (dBm), TX/RX rates, uptime.
- **Structured result:** `{ routerId, clients: [...], total, hasMore, offset, limit }`

### `manage_wifi_interface`

- **Type:** write (`destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: enable/disable/update), `interface`, `ssid` (optional string, required when action=update — enforce with `.superRefine()`), `dryRun`
- **Behavior:**
  - Look up interface by name from `interface/wifi`; throw `NOT_FOUND` if missing.
  - **enable/disable:** Set `disabled` flag via `update`. If already in desired state → `already_done`.
  - **update:** Look up the linked configuration name from the interface record. Update the `ssid` field on `interface/wifi/configuration/<id>`.
- **Structured result:** `{ action, interface, changes }` — changes lists before/after for each patched field.

## Phase 4 — WireGuard Tools

**New file:** `src/domain/tools/wireguard-tools.ts`  
**RouterOS paths:** `interface/wireguard`, `interface/wireguard/peers`

### `list_wireguard_interfaces`

- **Type:** read-only
- **Input:** `routerId`, `limit`, `offset`
- **Behavior:** Fetch from `interface/wireguard`. Fields: name, listen port, public key, MTU, running state.
- **Structured result:** `{ routerId, interfaces: [...], total, hasMore, offset, limit }`

### `manage_wireguard_interface`

- **Type:** write (`destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: add/remove), `name`, `listenPort` (optional), `mtu` (optional, default 1420), `comment` (optional), `dryRun`
- **Idempotency key:** `name`
- **add:** Look up by name.
  - Same name, same listen port → `already_exists`
  - Same name, different listen port → `CONFLICT`
  - Not found → create. RouterOS auto-generates the key pair; return the public key in structured result.
- **remove:** Not found → `not_found`. Found → delete.

### `list_wireguard_peers`

- **Type:** read-only
- **Input:** `routerId`, `interface` (optional filter), `limit`, `offset`
- **Behavior:** Fetch from `interface/wireguard/peers`. Filter client-side if `interface` is provided. Fields: interface, public key, endpoint address/port, allowed addresses, last handshake, TX/RX bytes.
- **Structured result:** `{ routerId, peers: [...], total, hasMore, offset, limit }`

### `manage_wireguard_peer`

- **Type:** write (`destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: add/update/remove), `interface`, `publicKey`, `endpointAddress` (optional), `endpointPort` (optional, 1–65535), `allowedAddresses` (optional string array of CIDRs), `presharedKey` (optional), `persistentKeepalive` (optional, seconds), `comment` (optional), `dryRun`
- **Idempotency key:** `publicKey` (unique across all interfaces)
- **add:** Look up by public key across `interface/wireguard/peers`.
  - Found → `already_exists`
  - Not found → create
- **update:** Look up by public key. Not found → `NOT_FOUND`. Found → patch only the fields that are explicitly provided (undefined = leave unchanged). Return before/after diff in structured result.
- **remove:** Look up by public key. Not found → `not_found`. Found → delete.

## Phase 5 — DNS Tools

**New file:** `src/domain/tools/dns-tools.ts`  
**RouterOS paths:** `ip/dns`, `ip/dns/static`

### `list_dns_entries`

- **Type:** read-only
- **Input:** `routerId`, `name` (optional substring filter), `limit`, `offset`
- **Behavior:** Fetch all from `ip/dns/static`. If `name` is provided, filter client-side with `includes()` (case-insensitive). Fields: name, type, address, TTL, disabled, comment.
- **Structured result:** `{ routerId, entries: [...], total, hasMore, offset, limit }`

### `manage_dns_entry`

- **Type:** write (`destructiveHint: false`, `idempotentHint: true`)
- **Input:** `routerId`, `action` (enum: add/remove), `name`, `address` (required for add, validated as IPv4), `ttl` (optional, seconds), `comment` (optional), `dryRun`
- **Record type:** hardcoded to `"A"` internally (scope limitation for this milestone)
- **Idempotency key:** `(name, type="A")`
- **add:** Look up existing by name + type `A`.
  - Same name, same address → `already_exists`
  - Same name, different address → `CONFLICT` with existing address in `details`
  - Not found → create
- **remove:** Look up by name + type. Not found → `not_found`. Found → delete.

### `get_dns_settings`

- **Type:** read-only
- **Input:** `routerId` only
- **Behavior:** Fetch single record from `ip/dns`. Fields: servers (upstream DNS array), `dynamic-servers`, `use-doh-server`, `verify-doh-cert`, `allow-remote-requests`, `max-udp-packet-size`, `query-server-timeout`, `query-total-timeout`, `max-concurrent-queries`, `max-concurrent-tcp-sessions`, `cache-size`, `cache-used`, `cache-max-ttl`.
- **Structured result:** `{ routerId, settings: { ...record } }`

## Testing approach

Every new tool file gets a corresponding test file in `test/unit/domain/tools/`. Test groups follow the project convention:

1. **metadata** — correct tool count and names in exported array, correct annotations
2. **input schema** — valid input with defaults, rejection of extra fields, rejection of invalid values
3. **handler — happy path** — successful create/list/remove
4. **handler — idempotency** — `already_exists`, `already_done`, `not_found`
5. **handler — conflict** — CONFLICT thrown with correct details
6. **handler — dry-run** — returns diff without calling client mutating methods

Infrastructure fix tests live in their respective existing test files (`circuit-breaker.test.ts`, `error-enricher.test.ts`, `router-registry.test.ts`).
