# v0.7 — Identity, Auth & Audit: Design Spec

**Date:** 2026-05-14  
**Milestone:** v0.7  
**Status:** Approved

---

## Overview

v0.7 establishes the trust boundary layer for MikroMCP before any further expansion of admin-level surfaces. Five concerns are in scope:

1. HTTP bearer token authentication
2. RBAC / identity enforcement
3. Audit log (dual-sink)
4. Confirmation middleware (two-step destructive gate)
5. Credential surface reduction (SSH/FTP adapters)

**Out of scope for v0.7:** HashiCorp Vault credential provider (slipped to v0.8).

---

## Architecture

Option B was selected: a dedicated `src/middleware/` layer with focused modules, an adapter factory for credential isolation, and an audit writer in `src/observability/`. The tool registry orchestrates all concerns in sequence; tool handlers remain unaware of auth logic.

## Section 1: Identity Config & Authentication

### Identity Config File

A new `config/identities.yaml` (separate from `routers.yaml`). Path controlled by `MIKROMCP_IDENTITIES_PATH` env var (default `config/identities.yaml`). Loaded and Zod-validated at startup — fast-fails with a descriptive message on bad config.

```yaml
identities:
  ci-pipeline:
    token: "$2b$12$..."           # bcrypt hash of the raw bearer token
    role: operator
    allowedRouters: ["edge-01", "edge-02"]  # empty list = all routers
    allowedToolPatterns: ["list_*", "ping", "traceroute"]  # empty list = all tools
  admin-ali:
    token: "$2b$12$..."
    role: admin
    allowedRouters: []
    allowedToolPatterns: []
```

**Token generation:** `openssl rand -hex 32` to generate; use a bcrypt tool such as `node -e "const b=require('bcryptjs');b.hash(process.argv[1],12).then(console.log)" <token>` or `htpasswd -bnBC 12 "" <token>` (Apache `htpasswd` with `-C 12` produces true bcrypt `$2y$` hashes). Raw tokens never appear in config.

**Roles:** `readonly` | `operator` | `admin` | `superadmin`. Role is a semantic label — enforcement is done by `allowedRouters` and `allowedToolPatterns`. Role is used only by the confirmation middleware to determine whether confirmation is required.

### `src/config/identity-registry.ts`

Mirrors `router-registry.ts`:
- Loads and parses `identities.yaml` with the `yaml` package
- Validates with a Zod schema (`.strict()`)
- Exposes `getIdentities(): IdentityConfig[]` and `findIdentityByToken(token: string): IdentityConfig | null` (bcrypt compare)
- Logs a startup warning if the file does not exist (identities.yaml is optional — stdio callers don't need it)

### `src/middleware/auth.ts`

Runs inside the HTTP transport request handler, before the MCP SDK receives the request.

1. Extracts `Authorization: Bearer <token>` header
2. Calls `findIdentityByToken(token)` — bcrypt compare against all identity hashes
3. On match: stores `Identity` in `AsyncLocalStorage` (same pattern as `correlation.ts`)
4. On no match or missing header: throws `PERMISSION_DENIED` with code `INVALID_TOKEN`

**stdio transport:** No token required. Identity is resolved from `MIKROMCP_STDIO_IDENTITY` env var (name of an identity in `identities.yaml`). If unset, a built-in `superadmin` identity is used with empty allow-lists (all access).

### New env vars

| Variable | Default | Notes |
|---|---|---|
| `MIKROMCP_IDENTITIES_PATH` | `config/identities.yaml` | Path to identity config |
| `MIKROMCP_STDIO_IDENTITY` | _(superadmin built-in)_ | Identity name for stdio transport |

## Section 2: RBAC Enforcement

### `src/middleware/authz.ts`

Runs at the start of each tool call in `tool-registry.ts`, before credentials are fetched.

**Router check:** If `identity.allowedRouters` is non-empty, the `routerId` param must be in the list. Throws `PERMISSION_DENIED` (code `ROUTER_NOT_ALLOWED`) if not.

**Tool check:** If `identity.allowedToolPatterns` is non-empty, the tool name must match at least one pattern. Patterns support `*` wildcard only (no full glob). Throws `PERMISSION_DENIED` (code `TOOL_NOT_ALLOWED`) if none match.

**Role** is not enforced as a separate gate here. It exists as a label for confirmation middleware and for human-readable audit records.

**ToolContext extension:** `identity: Identity` is added to `ToolContext` so tool handlers can read `context.identity.role` for logging. Tools must not perform their own auth checks.

**Cost:** The RBAC check runs before credentials are fetched and before the circuit breaker is touched — a denied call is cheap.

## Section 3: Confirmation Middleware

### `src/middleware/confirmation.ts`

Intercepts tool calls where both conditions are true:
- `toolDefinition.annotations.destructiveHint === true`
- `identity.role` is `readonly` or `operator`

Identities with role `admin` or `superadmin` bypass confirmation.

### First call (no confirmation token)

Middleware returns `APPROVAL_REQUIRED` before the handler runs:

```json
{
  "category": "APPROVAL_REQUIRED",
  "code": "CONFIRMATION_REQUIRED",
  "message": "This action is destructive. Re-submit with confirmationToken to proceed.",
  "details": {
    "confirmationToken": "a3f9...",
    "expiresAt": "2026-05-14T10:05:00Z",
    "tool": "manage_firewall_rule",
    "routerId": "edge-01",
    "action": "remove"
  }
}
```

**Token construction:** HMAC-SHA256 (keyed on `MIKROMCP_CONFIRMATION_SECRET` env var) over `{ tool, routerId, paramHash, identityId, expiresAt }`. TTL is 5 minutes. Tokens stored in an in-memory `Map<tokenHash, PendingConfirmation>` and evicted on use or expiry. `paramHash` is a SHA256 of the canonical JSON of the call params (excluding `confirmationToken`).

### Second call (with confirmation token)

Caller re-submits the identical tool call with `confirmationToken` added to params. Middleware:
1. Verifies HMAC signature
2. Checks expiry
3. Checks token has not been used (single-use)
4. Verifies `paramHash` matches the current call's params

On success: token consumed, execution proceeds normally.

### Schema injection

`confirmationToken` is an optional `z.string()` field injected into every destructive tool's input schema automatically by the tool registry. Tool definitions do not declare it themselves.

### New env var

| Variable | Default | Notes |
|---|---|---|
| `MIKROMCP_CONFIRMATION_SECRET` | _(required at startup if any identity has role `readonly` or `operator`)_ | HMAC signing key; server exits with error on startup if absent and such identities are configured |

## Section 4: Audit Log

### `src/observability/audit-log.ts`

Exports `auditLog(event: AuditEvent): void`. Two sinks:
- **Pino logger** — writes with `type: "audit"` field, using a dedicated `createLogger("audit")` instance
- **File sink** — if `MIKROMCP_AUDIT_LOG_PATH` is set, appends NDJSON lines via `fs.appendFileSync`. File opened once at module init. No buffering, no rotation (delegate to `logrotate`).

### AuditEvent type

```typescript
interface AuditEvent {
  type: "audit";
  ts: string;                           // ISO8601
  correlationId: string;
  identity: string;                     // identity name, never token
  role: string;
  tool: string;
  routerId: string;
  phase: "attempt" | "success" | "failure";
  params: Record<string, unknown>;      // credentials stripped before logging
  outcome?: string;                     // error code on failure
  durationMs?: number;                  // populated on success/failure
}
```

### When audit records are written

- **Only** for tools where `destructiveHint: true` or `readOnlyHint: false`
- `attempt` record: written before handler runs (ensures a trace even if process crashes)
- `success` record: written in the `finally` block on clean completion, with `durationMs`
- `failure` record: written in the `finally` block on error, with `outcome` (error code) and `durationMs`

Read-only tools are not audited to maintain signal-to-noise ratio.

### Credential stripping

Before writing `params`, an explicit redaction pass removes any field named `password`, `token`, `secret`, `authorization`, or `credentials` at any nesting depth.

### New env var

| Variable | Default | Notes |
|---|---|---|
| `MIKROMCP_AUDIT_LOG_PATH` | _(unset = file sink disabled)_ | Absolute path to audit NDJSON file |

## Section 5: Credential Surface Reduction

### `src/adapter/adapter-factory.ts`

Two factory functions:

```typescript
function createSshClient(routerConfig: RouterConfig, sshOptions: SshOptions): SshClient
function createFtpClient(routerConfig: RouterConfig): FtpClient
```

Both call `getCredentials(routerConfig)` internally. Credentials are embedded in the returned client's closure — not exposed as public properties.

### ToolContext changes

```typescript
// Removed
credentials: { username: string; password: string };

// Added
sshClient: SshClient;    // pre-configured, credentials in closure
ftpClient: FtpClient;    // pre-configured, credentials in closure
identity: Identity;      // from Section 2
```

### Tool handler changes

```typescript
// Before
const ssh = new SshClient(context.routerConfig, context.credentials, context.sshOptions);
const output = await ssh.execute(cmd);

// After
const output = await context.sshClient.execute(cmd);
```

FTP tools similarly update from inline credential objects to `context.ftpClient`.

### SshClient / FtpClient hardening

Credential fields (`username`, `password`) become `private` and are removed from all public interfaces and property accessors. The adapter internals remain unchanged.

### REST client

Already embeds credentials in its constructor (Base64 header built once at construction) — no change needed.

## Section 6: Execution Path End to End

### HTTP transport (`transports/http.ts`)

```
POST /mcp arrives
  → auth.ts: extract Bearer token → bcrypt compare → Identity → AsyncLocalStorage
  → MCP SDK handler
```

### Tool registry (`tool-registry.ts`), per tool call

```
1. authz.ts        → check allowedRouters + allowedToolPatterns
                     throws PERMISSION_DENIED if denied
2. confirmation.ts → if destructiveHint && role in [readonly, operator]:
                       check confirmationToken or return APPROVAL_REQUIRED
3. audit-log       → write "attempt" record
4. [existing]      → getCredentials → adapter-factory → build ToolContext
                     (sshClient, ftpClient, identity injected)
5. [existing]      → circuit breaker + retry wrapping
6. handler()       → tool executes
7. audit-log       → write "success" or "failure" record with durationMs
```

## Testing

Each new module gets a dedicated unit test file:

- `test/unit/middleware/auth.test.ts` — token extraction, bcrypt match/miss, stdio fallback
- `test/unit/middleware/authz.test.ts` — router allow/deny, tool pattern allow/deny, empty lists (all-access)
- `test/unit/middleware/confirmation.test.ts` — first call returns token, second call succeeds, expiry, single-use, param mismatch
- `test/unit/observability/audit-log.test.ts` — both sinks write, credentials stripped, attempt/success/failure phases
- `test/unit/config/identity-registry.test.ts` — valid YAML loads, bad YAML throws, unknown fields rejected

Existing tool tests updated to provide the new `ToolContext` shape (add `sshClient`, `ftpClient`, `identity` mocks; remove `credentials`).
