# MikroMCP v0.6 — Automation & System Management Design

**Date:** 2026-05-13
**Milestone:** v0.6
**Goal:** Scripts, Scheduler, Packages, Files, and Containers — the automation and lifecycle management layer.

---

## Overview

11 new tools across 5 subsystems, each in its own file following the established pattern. All write tools are idempotent and support dry-run. No new infrastructure required beyond FTP client support for `upload_file`.

---

## Subsystem 1: Scripts — `scripts-tools.ts`

**RouterOS path:** `/system/script`

### `list_scripts`
- **Input:** `routerId`, optional `name` (client-side substring filter)
- **Returns:** name, owner, run-count, last-started, source (script body), comment
- **Annotations:** `readOnlyHint: true`

### `manage_script`
- **Input:** `routerId`, `action: enum(add | update | remove)`, `name` (idempotency key), `source` (required on add/update), optional `comment`, optional `dontRequirePermissions: boolean`, `dryRun: boolean`
- **Behaviour:**
  - `add`: throws `CONFLICT` if a script with that name already exists
  - `update`: throws `NOT_FOUND` if script doesn't exist; replaces `source` and `comment`
  - `remove`: idempotent — already-removed returns `already_removed`, not an error
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`

### `run_script`
- **Input:** `routerId`, `name`
- **Behaviour:** looks up script by name → POST `/system/script/<id>/run`. Fire-and-forget — does not wait for completion or capture output.
- **Description note:** *"Script output is written to the router system log. Use `get_log` after calling this tool to see results."*
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: false`

## Subsystem 2: Scheduler — `scheduler-tools.ts`

**RouterOS path:** `/system/scheduler`

### `list_scheduled_jobs`
- **Input:** `routerId`, optional `name` (client-side filter)
- **Returns:** name, on-event, start-date, start-time, interval, run-count, disabled state
- **Annotations:** `readOnlyHint: true`

### `manage_scheduled_job`
- **Input:** `routerId`, `action: enum(add | update | remove | enable | disable)`, `name` (idempotency key), `onEvent` (required on add — script name or inline command), optional `startDate`, `startTime`, `interval` (RouterOS format e.g. `00:05:00`), `comment`, `dryRun: boolean`
- **Behaviour:**
  - `add`: throws `CONFLICT` if name already exists
  - `update`: throws `NOT_FOUND` if name doesn't exist; replaces provided fields
  - `remove`: idempotent — already-removed returns `already_removed`
  - `enable` / `disable`: throws `NOT_FOUND` if name doesn't exist; no-op if already in target state
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`

## Subsystem 3: Packages — `packages-tools.ts`

**RouterOS path:** `/system/package`

### `list_packages`
- **Input:** `routerId`, optional `name` (client-side filter)
- **Returns:** name, version, build-time, enabled state
- **Annotations:** `readOnlyHint: true`

### `manage_package`
- **Input:** `routerId`, `action: enum(enable | disable)`, `name` (idempotency key), `dryRun: boolean`
- **Behaviour:**
  - Throws `NOT_FOUND` if package doesn't exist
  - No-op if already in target state
- **Description note:** *"Package state changes take effect only after a router reboot — use the `reboot` tool to apply."*
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`

---

## Subsystem 4: Files — `files-tools.ts`

**RouterOS path:** `/file`

### `list_files`
- **Input:** `routerId`, optional `name` (client-side substring filter), optional `type` (e.g. `script`, `backup`)
- **Returns:** name, type, size, creation-time
- **Annotations:** `readOnlyHint: true`

### `get_file_content`
- **Input:** `routerId`, `name`
- **Behaviour:** GET `/file/<name>` content field. Throws `NOT_FOUND` if file doesn't exist.
- **Description note:** *"Only suitable for text files — binary files will return garbled content."*
- **Annotations:** `readOnlyHint: true`

### `upload_file`
- **Input:** `routerId`, `name` (target filename on router), `content` (string), `dryRun: boolean`
- **Behaviour:** Uploads via FTP using the router's existing credentials (username/password from the router registry — no additional credential fields). Overwrites if file already exists (RouterOS FTP behaviour). In dry-run mode, validates FTP connectivity without writing.
- **FTP client:** Node.js `basic-ftp` package (lightweight, Promise-based, no native deps)
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`

### FTP permission requirement (docs)
`config/routers.example.yaml` and README must note that the MikroTik user requires FTP permission:
```
/user set <name> group=<group>
/user group set <group> policy=...,ftp,...
```
This must be documented as a prerequisite for `upload_file`.

## Subsystem 5: Containers — `container-tools.ts`

**RouterOS path:** `/container`

### `list_containers`
- **Input:** `routerId`
- **Returns:** name, tag (image), status (`running`, `stopped`, `error`), interface, root-dir, mounts
- **Annotations:** `readOnlyHint: true`

### `manage_container`
- **Input:** `routerId`, `action: enum(create | start | stop | remove)`, `name` (idempotency key), and action-specific fields:
  - `create`: `remoteImage` (required, e.g. `alpine:latest`), `interface` (required — veth interface name), optional `rootDir`, `envlist` (RouterOS environment list name), `comment`
  - `start` / `stop` / `remove`: only `name` needed beyond `routerId`
  - `dryRun: boolean` on all actions
- **Behaviour:**
  - `create`: throws `CONFLICT` if container with that name already exists
  - `start`: throws `NOT_FOUND` if missing; no-op if already running
  - `stop`: throws `NOT_FOUND` if missing; no-op if already stopped
  - `remove`: throws `NOT_FOUND` if missing (explicit failure — removing a non-existent container is likely a mistake; intentionally non-idempotent unlike `manage_script` remove)
- **Description note:** *"Container creation requires a pre-configured veth interface in RouterOS. Use `create` to register and pull the image; use `start`/`stop` to manage lifecycle."*
- **Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: true`

## File structure

```
src/domain/tools/
  scripts-tools.ts        # list_scripts, manage_script, run_script
  scheduler-tools.ts      # list_scheduled_jobs, manage_scheduled_job
  packages-tools.ts       # list_packages, manage_package
  files-tools.ts          # list_files, get_file_content, upload_file
  container-tools.ts      # list_containers, manage_container
  index.ts                # add new tool arrays to allTools
```

New dependency: `basic-ftp` (npm) for `upload_file`.

---

## Error handling

Follows the existing pattern throughout:
- Handler body wrapped in `try/catch`
- `enrichError` called in catch
- `MikroMCPError` re-thrown as-is
- Domain errors (`NOT_FOUND`, `CONFLICT`, `VALIDATION`) thrown directly as `MikroMCPError`

---

## Testing

Each tool file gets a corresponding `test/unit/domain/tools/<name>.test.ts` following the existing pattern:
- `metadata` → `input schema` → `handler - <action>`
- Mock `routerClient` via `vi.fn()`
- Cover: happy path, idempotency (`already_exists`, `already_removed`, no-op), CONFLICT, NOT_FOUND, dry-run

FTP in `upload_file` is mocked at the `basic-ftp` client boundary.
