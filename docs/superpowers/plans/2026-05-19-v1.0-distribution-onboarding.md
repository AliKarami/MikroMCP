# MikroMCP v1.0.0 ŌĆö Implementation Plan

## Context

v0.9 shipped 77 tools and fleet operations. v1.0 is the distribution and ecosystem milestone ŌĆö it doesn't add router surfaces; it makes MikroMCP accessible and production-deployable. The five themes are:

1. A unified `mikromcp` CLI with `doctor` (diagnostics + setup wizard) and `init` (first-time onboarding)
2. Docker image published to Docker Hub + GHCR on every release tag
3. Standalone binaries for Linux/macOS/Windows via `pkg`, attached to GitHub Releases
4. Wiki onboarding guides for non-experts
5. Automated release CI that enforces the full artifact chain (binaries ŌåÆ Docker ŌåÆ GitHub Release ŌåÆ docs)

**Out of scope for v1.0:** Prometheus metrics, CHR integration test harness (deferred to v1.1).

### Task 2 ŌĆö `mikromcp doctor` command
**Goal:** Interactive health dashboard that checks every layer from config to router connectivity to Claude Desktop registration.

**File:** `src/cli/doctor.ts`

Checks (in order):
1. **Node.js version** ŌĆö `process.version` Ōēź 22; fail fast if not
2. **Config file** ŌĆö `MIKROMCP_CONFIG_PATH` set; file exists; parses as valid YAML; passes RouterRegistry schema validation (reuse `RouterRegistry` from `src/config/router-registry.ts`)
3. **Env vars** ŌĆö warn on missing `MIKROMCP_CONFIRMATION_SECRET` (HTTP transport), `ROUTER_<PREFIX>_USER/PASS` for each router; suggest writing a `.env` file if none found
4. **Router connectivity + permissions** ŌĆö for each router: connect with credentials from env; fetch `system/resource` (REST), then probe each permission tier:
   - **REST API** ŌĆö basic `GET system/resource` succeeds ŌåÆ Ō£ģ
   - **read policy** ŌĆö `GET ip/address` returns results ŌåÆ Ō£ģ (needed for most list tools)
   - **write policy** ŌĆö attempt `GET ip/firewall/filter` header check ŌåÆ infer from error pattern
   - **SSH** ŌĆö attempt TCP connect to port 22 (or configured SSH port) ŌåÆ Ō£ģ/ŌÜĀ’ĖÅ (needed for `ping`, `traceroute`, `torch`, `run_command`)
   - **FTP** ŌĆö attempt TCP connect to port 21 ŌåÆ Ō£ģ/ŌÜĀ’ĖÅ (needed for `upload_file`)
   - **reboot/sensitive** ŌĆö check identity role vs. tool categories that require `admin`/`superadmin`
   - Report: per-router table of Ō£ģ/ŌÜĀ’ĖÅ/ŌØī per capability tier with a one-line explanation of which tools are affected by any gap
5. **Identities config** ŌĆö if `MIKROMCP_IDENTITIES_PATH` set and file exists: validate schema; warn if no identity is configured for the current transport; note if all tokens are default/weak
6. **Claude Desktop detection** ŌĆö check `~/Library/Application Support/Claude/claude_desktop_config.json` (macOS) and `~/.config/Claude/claude_desktop_config.json` (Linux/Windows WSL); report whether `mikrotik-mcp-server` is present; if absent, print the JSON snippet to add
7. **Update check** ŌĆö fetch `https://registry.npmjs.org/mikromcp/latest` and compare `.version` to local `package.json`; warn if outdated with the upgrade command

Output format: colored rows (`Ō£ģ / ŌÜĀ’ĖÅ  / ŌØī`) via `chalk`, one line per check, followed by a summary section with fix suggestions for any failures.

**Tests:** `test/unit/cli/doctor.test.ts` ŌĆö mock `RouterRegistry`, `RouterOSRestClient`, `fetch`, `fs` (for Claude Desktop config); verify each check produces the correct status

### Task 3 ŌĆö `mikromcp init` wizard
**Goal:** Let a first-time user go from zero to a working `routers.yaml`, `identities.yaml`, `.env`, and registered Claude Desktop entry without touching any config file manually.

**File:** `src/cli/init.ts`

Wizard steps (using `@inquirer/prompts`):

**Router config:**
1. Router ID (the key in YAML, e.g. `core-01`)
2. Router host/IP
3. Port (default 80)
4. TLS: enabled (default **no** ŌĆö simpler for first-time users; if enabled, ask rejectUnauthorized, warn if false)
5. Credential env prefix (e.g. `ROUTER_CORE01`)
6. Tags (comma-separated, optional)
7. RouterOS version (default `7`)

**Identity / token:**
8. Ask whether to create an identity for HTTP transport (skip for stdio)
9. If yes: identity ID (e.g. `admin`), role (`superadmin`/`admin`/`operator`/`readonly`), allowed routers (default `*`), allowed tool patterns (default `*`) ŌĆö generate a secure random bearer token via `crypto.randomBytes(32).toString("hex")` and bcrypt-hash it for `identities.yaml`

**Env file:**
10. Ask whether to write env vars to `.env` (default yes); if yes, write `ROUTER_<PREFIX>_USER`, `ROUTER_<PREFIX>_PASS`, and optionally `MIKROMCP_CONFIRMATION_SECRET`, `MIKROMCP_TRANSPORT`; add `.env` to `.gitignore` if not already present; print a note to use `dotenv` or `source .env` before starting

**Config files:**
11. Write/append `config/routers.yaml`
12. Write/append `config/identities.yaml` (if identity was created)

**Claude Desktop:**
13. Offer to register with Claude Desktop ŌĆö patch `claude_desktop_config.json` with the `mcpServers` entry; back up the file first

**Summary:**
14. Print final checklist: env vars set (or `.env` written), files created, `npm start` command, `mikromcp doctor` for verification

**Tests:** `test/unit/cli/init.test.ts` ŌĆö mock `@inquirer/prompts`, `fs/promises`, `crypto`; verify YAML written correctly, `.env` written correctly, token generated and hashed, Claude Desktop JSON patched, backup created

### Task 4 ŌĆö Dockerfile + docker-compose
**Goal:** Multi-stage image, runnable with a single `docker run` or `docker compose up`.

**Files to create:**

```

WORKDIR /app
COPY package*.json ./

WORKDIR /app
```
